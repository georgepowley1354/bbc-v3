# Sprint Cadence

BBC runs on a loose biweekly sprint rhythm. It's not rigid Scrum — it's a framework for staying honest about what's in progress and what's actually getting done.

---

## Weekly Rhythm

### Monday — Planning (30 min)
- Review `project-bank.md` — what's active, what's stalled
- Check `brief-lab/` for any proposals needing follow-up
- Set the week's 3 priorities (no more than 3)
- Clear any client messages from the weekend

### Tuesday – Thursday — Build Days
- Deep work blocks: 9am-12pm and 2pm-5pm
- Client communication: check and respond between 9-11am only
- No new feature additions mid-sprint — capture in a change order note instead
- If blocked, note why in the project file (don't spin for more than 2 hours)

### Friday — Wrap and Update (1 hour)
- Send status updates to any active clients (see `client-reporting.md`)
- Update `project-bank.md` with milestone progress
- Log any new leads in `lead-capture.md`
- Note 1 thing that went well and 1 thing to improve next week

---

## Biweekly Sprint Structure

### Sprint Start (every other Monday)
- Define sprint goal: what ships or reaches a milestone this sprint?
- Break into 2-3 tasks per active project
- Note any dependencies (waiting on client assets, approvals, etc.)

### Sprint End (every other Friday)
- What shipped? What didn't?
- What's blocking progress?
- Does the timeline need updating?
- Send client updates for anything that moved

---

## Project Phase Timings

| Phase | Typical Duration | What Can Slip It |
|-------|-----------------|-----------------|
| Discovery | 3-5 business days | Client availability |
| Design | 5-10 business days | Revision rounds |
| Build | 5-15 business days | Scope additions, CMS complexity |
| Review | 3-5 business days | Client response time |
| Launch | 1-2 business days | DNS propagation |

Build in 20% buffer on all client-facing timelines. Under-promise, over-deliver.

---

## Managing Multiple Projects

When running 2+ active projects simultaneously:
- Assign each project a "primary day" (e.g., kris = Tuesday, whatsshakin = Thursday)
- Avoid context-switching mid-task — finish a component before switching clients
- Keep `project-bank.md` updated so you always know the status without re-reading files

**Max active projects at one time:** 3 (design/build phase). Retainer clients don't count unless they have active change orders.
